ü§ñ Non-Human Identity Interactions

Non-human identities (applications, services, and automated processes) require secure authentication to access Key Vault resources. The following guidance covers common scenarios and recommended authentication patterns:

üîê Service Principals (Application Identities)

  • Use Case: Web applications, APIs, background services, and CI/CD pipelines
  • Recommended Approach: Use certificate-based authentication instead of client secrets
  • Security Benefits: Certificate rotation is automated, no secrets in configuration, stronger authentication
  • Implementation:
    • Store certificates in Key Vault and use them for service principal authentication
    • Implement certificate auto-rotation using Azure Automation or Azure Functions
    • Use Azure AD managed identities where possible instead of service principals

‚öôÔ∏è Automation & DevOps

  • Use Case: Azure DevOps pipelines, GitHub Actions, Jenkins, and other CI/CD systems
  • Recommended Approach: Use workload identity federation or managed identities
  • Security Benefits: No long-lived secrets, automatic token management, reduced attack surface
  • Implementation:
    • Azure DevOps: Use service connections with workload identity federation
    • GitHub Actions: Configure OpenID Connect (OIDC) for Azure
    • Grant minimal required permissions using Azure RBAC roles

üîÑ Batch Processing & Scheduled Jobs

  • Use Case: Azure Functions (timer-triggered), Azure Automation runbooks, Azure Logic Apps
  • Recommended Approach: System-assigned managed identities with minimal privilege
  • Security Benefits: No credential management, automatic scaling, built-in rotation
  • Implementation:
    • Enable managed identity on the Azure resource
    • Grant Key Vault RBAC roles (Key Vault Secrets User, Key Vault Crypto User, etc.)
    • Use Azure Key Vault references in App Settings for Azure Functions
